# Crawfish v0.1 Plan

Canonical terminology is defined in [`glossary.md`](glossary.md).
Forward-looking design principles are defined in [`philosophy.md`](philosophy.md).

## Release Goal

Ship an alpha runtime that proves Crawfish is not just another orchestration wrapper. v0.1 must demonstrate:

- lifecycle-managed agents
- durable action execution
- execution contract enforcement
- MCP-based tool integration
- restart recovery from checkpoints
- continuity under severe model or harness outage
- explicit same-device governance primitives for foreign-owner encounters
- inspectable operational state

The alpha must do this through one consistent public story: the engineering and operations swarm made up of `repo_indexer`, `repo_reviewer`, `ci_triage`, `workspace_editor`, `incident_enricher`, and `task_planner`.

Implementation baseline for this alpha is Rust-first, not Rust-only: the supervisor daemon, CLI, state machine logic, contract compiler, governance logic, and SQLite persistence live in the Rust workspace, while isolated edge bridges may exist outside the runtime spine behind the adapter boundary.

The alpha does not need to ship every interoperability plane. It needs to prove the runtime philosophy: bounded agency, contract enforcement, lifecycle control, recoverable execution, continuity when the best reasoning surface disappears, and governance when agents from different owners meet on the same machine. ACP-compatible harness routing is strategically important, but it is a deliberate P1 expansion rather than a reason to bloat P0.

The product must remain standalone-first even as it leans into ecosystem leverage. The first external ecosystem priority after P0 is bidirectional OpenClaw interop, not because Crawfish should become an OpenClaw feature, but because OpenClaw is an important adjacent entry point and execution surface.

## Public Status And Support Policy

Crawfish is public and maintained seriously, but this release line is still **alpha**.

- version posture: `0.x` / `alpha`
- implementation baseline: Rust-first
- supported runtime environments: macOS and Linux
- supported MCP transport in the current codebase: SSE only

The following surfaces are public but unstable during alpha:

- CLI
- `Crawfish.toml` and agent manifests
- local UDS HTTP API
- Rust workspace crates

Breaking alpha changes are allowed. Every user-visible break must include:

- a changelog entry
- README or spec updates
- a migration note when the break affects real users or example workflows

## Current Implemented Slice

The current Rust workspace already proves one concrete alpha slice:

- `repo_indexer` produces a typed `repo_index.json` artifact from a local workspace.
- `repo_reviewer` consumes repository context and emits `review_findings.json` plus `review_summary.md`.
- `ci_triage` emits `ci_triage.json` plus `ci_triage_summary.md` from either direct log input or an SSE MCP fetch path.
- `incident_enricher` emits `incident_enrichment.json` plus `incident_summary.md` from local logs and optional service topology input.
- `workspace_editor` executes approval-gated `workspace.patch.apply` with deterministic local file operations.
- deterministic executors emit checkpoint metadata that can be inspected and reused during restart recovery.
- `inspect` already exposes artifact refs, checkpoint refs, recovery stage, encounter metadata, grant details, lease detail, blocked reason, terminal code, lock detail, and external refs.

## P0 Scope

Only the following capabilities are in P0:

1. `crawfishd` supervisor daemon
2. lifecycle manager
3. action execution
4. execution contract v1
5. MCP integration
6. checkpoint and resume
7. observability baseline
8. security baseline

Continuity baseline is delivered through these same P0 systems, especially lifecycle, action execution, recovery policy, and inspection. It is not a separate subsystem bolted on later.
Governance baseline is also delivered through these same P0 systems: manifests, action metadata, inspection, and policy validation must all understand owner, trust domain, encounter policy, and future lease-based execution.

## P1 Scope

- OpenClaw inbound Gateway RPC bridge, with plugin-tool UX deferred until after the RPC path is stable
- OpenClaw outbound Gateway adapter for agent execution
- A2A outbound remote-agent delegation for treaty-governed `task.plan`
- verified execution strategies beyond `task.plan`
- deterministic verifiers for proposal, investigation, and mutation-heavy actions
- ACP-compatible harness adapters
- ACPX-like harness router or broker integration pattern
- additional local harness adapters beyond Claude Code and Codex
- visual swarm dashboard
- multi-host supervisor and leased worker model
- context compression and memory compaction hooks
- richer evaluator classes, prompt/strategy comparison, and alert policies tied to quality policy
- broader federation governance packs beyond static local config, treaty negotiation, and richer remote-agent escalation semantics
- remote admissibility continuation and same-action follow-up re-delegation beyond the current `task.plan`-only slice

## P2 Scope

- marketplace and reusable capability packs
- policy DSL and simulation mode
- advanced multi-tenant control plane
- remote governance treaties and federation policy packs
- additional transport plugins
- adaptive scheduling based on historical cost and success curves
- controlled self-evolution tooling for routing, thresholds, and rule proposals

## Out Of Scope For v0.1

- consumer assistant experiences
- visual-first builders
- cluster orchestration beyond the single-host reference architecture
- broader OpenClaw production semantics such as sticky sessions, openclaw-managed workspaces, or action-level cancel and remote resume
- A2A inbound server behavior, multi-hop delegation, mutation over A2A, or secret forwarding
- reputation systems, marketplaces, or dispute resolution between owners
- enterprise multi-tenancy beyond foundational propagation rules
- unrestricted runtime self-modification

## Hero Demo

The alpha demo must use the same swarm across docs and demos:

- `repo_indexer`
- `repo_reviewer`
- `ci_triage`
- `workspace_editor`
- `incident_enricher`
- `task_planner`

### Demo Narrative

1. The supervisor boots the swarm in dependency order.
2. A pull request review action enters the system with an explicit execution contract.
3. `repo_reviewer` relies on `repo_indexer` context and MCP-based GitHub or filesystem tools.
4. A dependency fault or liveliness miss forces a degraded profile rather than an uncontrolled crash.
5. A mutating local workspace step is held in `awaiting_approval`, then either approved into a lease-bound action or rejected with an audit trail.
6. A supervisor restart recovers the action from its last checkpoint.
7. A total external model outage contracts the swarm into deterministic continuity or store-and-forward behavior instead of opaque failure.
8. A same-device foreign-owner encounter is classified, inspected, and denied or escalated unless explicit policy and consent would allow it.

P1 extends the same narrative in two directions:

- an OpenClaw session or plugin can submit work into Crawfish and inspect long-running swarm actions
- selected proposal or high-context actions, such as `task.plan`, can route into local harnesses, treaty-governed remote A2A agents, or OpenClaw and still run under the same verified execution strategy when deterministic proof is required

Likely next directions after this slice are:

- richer evaluator classes over remote-agent outcomes
- broader federation governance packs beyond static local config, treaty negotiation, and richer remote-agent escalation semantics

## Acceptance Criteria

### 1. Supervisor Daemon

- Given a valid swarm manifest, `crawfish run` starts one supervisor and all declared agents.
- `crawfish status` shows desired state, observed state, owner, trust domain, health, degradation profile, and continuity mode for every agent.
- `crawfish drain` stops new action assignment and eventually moves active agents to `Inactive` or `Finalized`.

### 2. Lifecycle Manager

- Given an agent dependency graph, agents activate in dependency order.
- If a required dependency becomes unavailable but a degraded profile exists, dependent agents move to `Degraded` instead of crashing.
- If no approved external model or harness route remains for an action class, the affected agent enters an explicit continuity mode rather than silently retrying forever.
- If a hard safety policy becomes unsatisfied, the affected agent moves to `Failed`.
- The alpha must support at least `read_only`, `dependency_isolation`, and `budget_guard` degraded profiles as explicit, inspectable runtime modes.

### 3. Action Execution

- Every submitted action gets a stable id, a stored phase, feedback events, and a terminal record.
- Long-running actions can enter `blocked` or `awaiting_approval` without losing inspectability.
- When agentic execution is unavailable, an action follows its recovery policy into deterministic execution, store-and-forward, or human handoff rather than disappearing into retry loops.
- Cancelling an action records a terminal state even when an underlying tool does not cancel cleanly.
- `workspace.patch.apply` is supported as a local deterministic mutation path and only executes after approval and lease validation.

### 4. Execution Contract v1

- The runtime compiles contracts in the order `org defaults -> agent defaults -> action overrides`.
- Hard policy violations are rejected before execution starts.
- Soft policy pressure, such as provider instability or budget pressure, may trigger a deterministic degraded profile or fallback route.

### 5. Governance Baseline

- Manifests and actions can carry owner, trust-domain, and encounter-policy metadata.
- `policy validate` can explain whether a same-device foreign-owner request would be denied, require consent, or be leasable under current policy.
- Cross-owner mutable workspace access is denied by default unless explicit encounter policy would later permit grant and lease issuance.
- Same-owner local read-only work can be leased automatically; approval-gated mutation creates grants, leases, and audit receipts through operator flow.

### 6. MCP Integration

- Agents can discover MCP tools through configured bindings.
- Tool invocations are constrained by compiled tool scope.
- MCP tool failures are surfaced with action id, capability namespace, and failure taxonomy.
- v0.1 transport support is SSE-only. The reference hero path uses MCP to fetch remote CI material for `ci_triage`, while keeping the final classification deterministic and local.

### 7. Checkpoint And Resume

- The runtime checkpoints after model turns, mutating tool calls, and explicit yield points.
- After supervisor restart, resumable actions continue from the last valid checkpoint.
- If checkpoint recovery is impossible, the recovery policy determines replay, dead-letter, or failure.

### 8. Observability Baseline

- Every action and lifecycle transition emits structured log records and traceable identifiers.
- `crawfish inspect <action-id>` shows action phase, compiled contract, encounter state, grant or lease refs, assigned agent, dependencies, checkpoint reference, and terminal or current failure reason.
- `crawfish action events <action-id>` returns the operator timeline for admission, approval, lease, execution, checkpoint, artifact, recovery, and terminal transitions.
- `crawfish action trace <action-id>` returns the trace bundle for one action, including external refs, enforcement records, and policy incidents.
- `crawfish action evals <action-id>` returns deterministic evaluations and verification-linked scorecards.
- `crawfish review list` and `crawfish review resolve` expose the operator review queue without mutating prior action history.
- `crawfish eval dataset list`, `eval dataset show`, `eval run`, and `eval run-status` expose dataset-backed evaluation ops without polluting production operator state.
- `crawfish alert list` and `alert ack` expose doctrine and evaluation escalation signals with explicit acknowledgment state.
- `crawfish inspect <agent-id>` shows desired state, observed state, owner, trust domain, health summary, recent transitions, degradation profile, continuity mode, recent repair actions, and recovery condition.

### 9. Security Baseline

- Mutating tools cannot execute unless a policy explicitly allows the mutation mode.
- Approval-gated steps move the action to `awaiting_approval` until approval or expiry.
- Approved mutation steps issue `ConsentGrant` and `CapabilityLease`; revoked leases fail pending work before mutation commit.
- Workspace mutation under `lock_mode = file` must either acquire a workspace-scoped file lock or enter `blocked` with an explicit lock-conflict reason.
- Secret references are resolved only for adapters and scopes allowed by the compiled contract.
- Same-device foreign-owner access to workspace, memory, secret, or mutating capability is denied by default unless future encounter policy and lease rules allow it.

## Required Test Scenarios

### Dependency Failure And Degradation

- Start the hero swarm with `repo_indexer` healthy.
- Submit a review action that depends on index data.
- Simulate `repo_indexer` liveliness failure.
- Verify `repo_reviewer` moves to `Degraded` using `dependency_isolation` or equivalent profile.
- Verify new actions requiring the missing dependency are blocked or downgraded according to policy, not silently routed unsafely.

### Budget Guard Degradation

- Submit actions that move the swarm near the configured budget threshold.
- Verify the runtime contracts execution to a cheaper route or less expensive profile.
- Verify work that cannot satisfy the degraded budget policy is blocked or deferred rather than executed optimistically.

### Total Model Or Network Outage Continuity

- Simulate loss of every approved model, harness, or remote execution route for a chosen action class.
- Verify the affected agent enters `deterministic_only`, `store_and_forward`, or `human_handoff` according to policy.
- Verify deterministic continuity work still produces inspectable outputs where available.
- Verify operator surfaces clearly explain why the runtime contracted and what would restore full service.

### Restart Recovery

- Submit a long-running review or triage action.
- Force a checkpoint after at least one model turn and one tool call.
- Restart the supervisor.
- Verify the action resumes from the stored checkpoint rather than restarting from scratch.

### Hard Policy Interception

- Submit an action override that exceeds allowed budget or expands tool scope beyond organization policy.
- Verify the action is rejected before entering `running`.
- Verify the rejection reason is surfaced through `policy validate` and `inspect`.

### Mutating Tool Approval Gate

- Submit an action that proposes a write operation, such as a workspace edit or external ticket update.
- Verify the action enters `awaiting_approval` before the mutating call executes.
- Verify approval issues grant and lease state, rejection terminates it, and lease revocation blocks later mutation commit.

### CLI Inspection Path

- Verify `status` shows the full swarm at a glance.
- Verify `inspect <agent-id>` explains why an agent is degraded, blocked, or failed.
- Verify `inspect <action-id>` shows the compiled contract, current phase, dependency summary, and checkpoint reference.

### Same-Device Foreign-Owner Governance

- Define two agents on the same machine with different owners.
- Verify policy validation treats the encounter as `same_device_foreign_owner`.
- Verify cross-owner mutation or secret access is denied or escalated rather than allowed implicitly.
- Verify inspection can explain which governance rule produced the decision.

## Planned P1 Validation Scenarios

### OpenClaw Inbound Path

- Submit a Crawfish action from an OpenClaw tool, plugin, or Gateway RPC method.
- Verify Crawfish returns a stable action id.
- Verify subsequent OpenClaw inspection calls can retrieve phase, contract, degradation profile, and continuity mode through Crawfish APIs.
- Verify unmapped callers fail before action creation and do not leave durable action records behind.

### OpenClaw Outbound Path

- Submit a proposal-oriented action whose selected execution surface is `openclaw`.
- Verify Crawfish starts the action through the OpenClaw Gateway `agent` and `agent.wait` flow.
- Verify OpenClaw stream events map into Crawfish action feedback and external run ids.
- Verify unsupported `sticky` sessions and `openclaw_managed` workspace mode fail before dispatch.

### Verified Execution Strategy

- Submit a `task.plan` action that uses `verify_loop`.
- Verify the action runs at least two iterations when the first deterministic verification fails.
- Verify structured failure feedback is passed into the next iteration.
- Verify passing verification completes the action; exhausted stop budget moves it to `human_handoff` or `failed`.

### Continuity Under OpenClaw Outage

- Simulate OpenClaw Gateway unavailability for an action class that prefers `openclaw`.
- Verify Crawfish does not retry indefinitely.
- Verify the runtime falls back to deterministic planning when the contract fallback chain permits it.
- Verify otherwise the runtime contracts into `store_and_forward` or another configured continuity mode.
- Verify `inspect` and `events` explain the trigger, current continuity behavior, and restoration condition.

### Consent-To-Lease Flow

- Simulate a cross-owner request that is allowed only with explicit consent.
- Verify a `ConsentGrant` is created before a `CapabilityLease`.
- Verify an expired or revoked lease prevents future execution attempts.

### Same-Device Resource Arbitration

- Create two different-owner agents contending for the same mutable workspace or capability.
- Verify the runtime does not resolve the conflict by arrival order alone.
- Verify the outcome is explainable through owner, trust-domain, and encounter policy rules.

### OpenClaw Governance Path

- Map an inbound OpenClaw caller to owner, trust domain, and session scope before action admission.
- Verify OpenClaw outbound execution consumes a lease-bound scope and cannot exceed Crawfish policy.
- Verify audit and inspection surfaces can explain both the encounter decision and the execution outcome.

### Auditability

- Every cross-owner interaction should leave an `EncounterRecord` and `AuditReceipt`.
- Verify `inspect` can explain why access was allowed, denied, downgraded, expired, or revoked.

## Workstreams

### Workstream 1: Supervisor And Lifecycle

- reconcile desired state against observed state
- implement lifecycle transitions and health probes
- implement drain and recovery orchestration

### Workstream 2: Action Engine

- persist actions, feedback, and terminal state
- implement action phase transitions
- connect cancellation, blocking, and approval waiting
- support deterministic continuity executors through the same action model
- add execution strategy handling for `single_pass` and `verify_loop`

### Workstream 3: Contract Compiler And Policy

- define contract data model
- implement merge precedence and hard versus soft validation
- derive fallback and degradation behavior
- derive continuity behavior for total route loss, offline queueing, and human handoff
- define governance precedence for owner, trust domain, and encounter policy

### Workstream 4: MCP And Execution Adapters

- establish MCP client session management
- bind normalized capabilities to tool invocations
- record adapter-level failures and scopes
- keep the implemented OpenClaw inbound bridge thin so future plugin or outbound work does not change the action model
- design ACP-compatible harness adapter interfaces so P1 can add Codex, Claude Code, Gemini CLI, or ACPX-like brokers without changing the action model

### Workstream 5: State, Telemetry, And Inspection

- checkpoint action progress and worker state
- emit structured logs and traces
- implement `status`, `inspect`, and `policy validate`
- record repair attempts, continuity activations, and restoration outcomes
- record external run ids and verify-loop iteration state for verification-sensitive actions
- record encounter decisions, grant refs, lease refs, and audit receipt refs
