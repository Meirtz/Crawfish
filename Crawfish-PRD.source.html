Executive Summary

Crawfish is a production agent runtime, not another prompt wrapper or graph editor. It is designed for teams that need always-on agents to run with declared operational constraints: deadlines, budgets, tool permissions, approval gates, health checks, and orderly lifecycle transitions.

The current agent ecosystem is strong at three layers and weak at one critical layer. It is strong at model and tool APIs, strong at workflow durability, and increasingly strong at protocol interoperability. It is weak at runtime operations: how fleets of agents start, degrade, recover, pause, drain, and resume under real production conditions. That missing layer is where Crawfish lives.

Crawfish combines four ideas into one coherent runtime:

  • Managed lifecycle: agents are supervised resources with desired state, observed state, health, dependency ordering, and degraded modes.
  • Action-oriented execution: long-running work is modeled as a durable task object with feedback, cancellation, checkpointing, and resumability.
  • Execution contracts: each task carries explicit delivery, budget, safety, and routing constraints that are enforced by the runtime rather than left to prompt text.
  • Protocol-native integration: MCP is the default tool plane, A2A is the inter-agent plane, and local subprocess adapters support coding and automation workloads.
Crawfish is to agent operations what ROS2 is to robot operations: a reliability-first runtime that turns orchestration from ad hoc control flow into a managed system.

The beachhead is internal agent fleets for engineering, operations, and enterprise automation: systems that must stay up, stay within budget, and fail safely. The product is open-source core, local-first in developer experience, and production-oriented in its architecture.

1. Product Category And Positioning

1.1 Category Definition

Crawfish defines a category between agent SDKs and durable workflow engines: lifecycle-managed agent runtime.

This category exists because production agents are neither simple request handlers nor generic workflows. They are stateful, tool-using, failure-prone, budget-sensitive software actors that need an operational control plane.

1.2 Positioning Statement

For platform engineers and AI product teams running always-on or high-value agent systems, Crawfish is the runtime that adds lifecycle control, contract enforcement, and operational resilience. Unlike lightweight agent SDKs, Crawfish does not stop at tool calls and handoffs. Unlike workflow engines, Crawfish is model-native, protocol-native, and agent-native.

2. Market White Space And Competitive Gap

The market is crowded with frameworks, but most products cluster around only part of the real problem. The white space is not "more orchestration." The white space is operational control for agent fleets.

2.1 The Product Gap

Today, a team that wants a production agent system typically stitches together an agent SDK, an orchestration library, an event bus, tracing, a scheduler, secret handling, and a custom supervisor. That architecture works for prototypes but breaks down operationally. There is no consistent object for an agent's desired state, no standard degraded mode, no unified contract for budget and safety policy, and no clean bridge between tool protocols and agent protocols.

Crawfish occupies that gap. It is the runtime layer that sits between application logic and infrastructure plumbing.

2.2 Why This Gap Is Defensible

  • Lifecycle and control-plane concepts are cross-cutting. They are difficult to retrofit into request-scoped SDKs without breaking their simplicity model.
  • Execution contracts touch scheduling, routing, observability, security, and developer experience at once. This is architecture, not a helper library.
  • Protocol interoperability alone is not enough. The real challenge is policy-preserving interoperability: tool scope, deadlines, approvals, and failure semantics must survive cross-protocol boundaries.
  • Always-on systems need a reconciliation loop, not just a while-loop.

  1. Engineering and repo automation: code review, dependency updates, CI triage, flaky test isolation, release prep.
  2. Ops and incident automation: alert enrichment, runbook execution, remediation proposals, ticket generation.
  3. Back-office workflows: document intake, approval routing, SLA-governed email and queue processing.

4. Product Principles

  • Keep the agent loop simple: model thinking stays close to the agent; operational complexity lives in the runtime.
  • Policies over prompt text: budgets, tool scopes, deadlines, and approvals are machine-enforced contracts.
  • Graceful degradation beats binary failure: cheaper model, narrower tool scope, lower background priority, and delayed retries should come before hard stop whenever safe.
  • Desired state over imperative orchestration: the system should reconcile toward a declared target state.
  • Protocol-native, vendor-neutral: tools, remote agents, and transports should be replaceable without rewriting agent logic.
  • Developer-local first, production-ready second: single-host mode must be easy; cluster mode must not require a rewrite.
  • Observability is part of the product surface: cost, latency, failure reasons, and transitions are first-class outputs.

5.2 What Makes Crawfish Technically Different

  • It separates the control plane from the inference plane. Models decide task content; the runtime decides whether the task may run, on which agent, with which contract, and in which degradation profile.
  • It treats "QoS" as part of a broader execution contract. Transport semantics, budget policy, safety policy, and fallback rules live in one compiled plan.
  • It unifies tool and agent interoperability. MCP, A2A, and local subprocess adapters all feed a normalized capability model.
  • It is local-first without being local-only. The same programming model supports single-host mode and clustered mode.

6.1 Runtime Topology

6.2 Control Plane vs Execution Plane

Plane Responsibilities Non-responsibilities
Control plane Desired state, agent placement, policy compilation, drain and recover, dependency graph, admission control, lease management It does not generate task content or reason over user prompts
Execution plane Model calls, tool calls, local state mutation, checkpointing, feedback streaming, result artifacts It does not decide org-wide policy or fleet-level topology

This split is important. Many current frameworks mix scheduling logic, business logic, and model logic inside one loop. Crawfish keeps the loop simple by moving operational responsibility to the runtime.

6.3 Managed Lifecycle And Reconciliation

Every agent instance is managed by a reconciliation loop. The runtime continuously compares desired state to observed state and takes corrective action. This draws from ROS2 managed nodes and from modern control-plane design, but adapts both to model- and tool-driven workloads.

State Meaning Typical entry trigger Typical exit trigger
Unconfigured Manifest loaded, runtime not yet initialized Definition created or process discovered Configuration begins
Configuring Model adapters, tool sessions, secrets, and dependencies are being prepared Supervisor begins configure transition Configuration succeeds or fails
Inactive Ready but not accepting new work Configuration succeeds Activation or finalization
Activating Warm-up, health probe, cache restore, session restore Supervisor issues activate Active or Degraded
Active Accepting tasks under normal policy profile Health checks and dependencies are green Degraded, Draining, Failed
Degraded Agent stays online with reduced capability or higher restrictions Budget pressure, provider instability, repeated soft failures, dependency impairment Recovery, Draining, Failed
Draining No new tasks accepted; existing work completes or checkpoints Deploy, shutdown, dependency maintenance, rolling restart Inactive or Finalized
Failed Runtime cannot safely continue under current policy Hard policy violation, repeated crash loop, unrecoverable dependency loss Reconfigure, manual intervention, Finalized
Finalized Process decommissioned and resources released Delete or permanent shutdown None

The Degraded state is a key product differentiator. Instead of binary "healthy / unhealthy" logic, Crawfish lets a fleet preserve partial service safely. Example degradation profiles:

  • cost_saver: switch to cheaper model class, reduce background polling, lower context retention.
  • read_only: disable mutating tools and force proposal-only outputs.
  • latency_guard: shorten tool chains, cap retries, prefer cached context.
  • dependency_isolation: reject tasks that require impaired upstream agents while continuing independent work.

6.4 Task Model: Actions, Not Fire-And-Forget Calls

Long-running work is represented as a durable task action. This is the primary execution object in Crawfish. A task is accepted, scheduled, checkpointed, observed, and eventually resolved as a result artifact.

Field group Examples Why it exists
Identity task_id, tenant_id, requester, correlation_id, idempotency_key Supports traceability, deduplication, and audit
Goal intent, capability, input payload, context references, expected artifact types Makes work routable and inspectable
Contract deadline_ms, max_cost_usd, approval_policy, tool_scope, quality_class, retry_budget Moves runtime constraints out of prompt text
Scheduling priority, earliest_start, affinity, dependency set, preemption class Allows policy-aware scheduling and backpressure
Runtime state phase, heartbeat_at, checkpoint_ref, assigned_agent, blocked_reason Supports inspection and recovery
Outputs feedback events, trace refs, artifacts, final status, failure taxonomy Provides durable results and operational analysis

Task phases should include accepted, running, blocked, awaiting_approval, cancelling, completed, failed, and expired.

Cancellation is not best-effort theater. The runtime must propagate cancellation through model streams, tool sessions, and child actions, then persist the terminal state.

6.5 Communication Patterns

Pattern Purpose Semantics Example
Topics Fleet events and notifications Many-to-many, asynchronous, optionally durable Agent state changed, artifact created, budget threshold crossed
Services Synchronous control and data queries Request-response, short-lived, typed Inspect agent, fetch checkpoint, validate policy
Actions Long-running execution Goal, feedback, result, cancellation, checkpointing Review PR, enrich alert, research vendor, update dependency tree

The Action pattern is where Crawfish gains most of its product leverage. It gives the runtime a durable unit of work that can be scheduled, observed, paused, retried, or escalated without inventing custom control flow for every use case.

6.6 Execution Contracts

"QoS" is useful inspiration, but the user-facing abstraction should be broader and more intuitive: Execution Contract. A contract is compiled from organization defaults, agent manifest defaults, and per-task overrides.

Contract layer Fields Behavior
Delivery contract deadline_ms, freshness_ttl_ms, required_ack, liveliness_window_ms Defines time and delivery expectations
Execution contract max_cost_usd, max_tokens, model_class, fallback_chain, retry_budget Constrains model and compute behavior
Safety contract tool_scope, approval_policy, mutation_mode, data_zone, secret_policy Constrains what the task may touch
Quality contract quality_class, evaluation_hook, minimum_confidence, human_review_rule Controls escalation and review
Recovery contract checkpoint_interval, resumability, fallback_behavior, dead_letter_policy Controls restart and failure semantics

Two policy types are especially important:

  • Hard policies: must never be violated. Example: no write tools without approval, no crossing tenant boundary, no budget overspend.
  • Soft policies: the runtime may degrade or reroute before declaring failure. Example: target latency, preferred provider, preferred model class.

This distinction is a major product advantage. It lets the runtime stay operational under pressure without hiding real violations.

6.7 Policy Compiler And Execution Plan

The runtime should not consult an LLM to decide scheduling or policy enforcement. Instead, a deterministic policy compiler produces an Execution Plan from three inputs:

  1. Organization rules
  2. Agent manifest defaults
  3. Task-level overrides

The compiled plan determines model routing, allowed tools, approval requirements, retry and fallback behavior, checkpoint cadence, and observability tags. This gives users predictability and auditability.

6.8 Scheduler, Concurrency, And Backpressure

Crawfish should use a scheduler that is explicit about priority, deadline, and resource limits. The runtime must assume that agents are expensive and that tool calls can block unpredictably.

  • Priority bands: control traffic, user-facing tasks, background maintenance, best-effort cron work.
  • Deadline-aware ordering: within a band, earlier deadlines outrank later ones.
  • Mutually exclusive groups: preserve state safety for tasks sharing a mutable workspace or memory scope.
  • Reentrant groups: allow parallel execution for stateless or read-only work.
  • Admission control: reject or defer tasks when budget, token rate, or tool session capacity is exhausted.
  • Lease-based ownership: each active task has a lease; loss of lease triggers recovery rather than duplicate execution.

6.9 State, Checkpointing, And Recovery

Crawfish needs durable execution, but it should not force users to learn a heavyweight workflow DSL. The right model is checkpointed agent turns.

  • Checkpoint after each model turn, after each mutating tool call, and at explicit yield points for long tool operations.
  • Persist enough state to resume safely: task phase, agent memory refs, tool outputs, pending approvals, artifact refs, and trace linkage.
  • Store artifacts separately from state snapshots so large outputs do not bloat recovery records.
  • On restart, the supervisor reconstructs the action graph, reassigns resumable tasks, and marks non-resumable ones for replay or escalation based on policy.

Developer mode should default to SQLite-backed state. Production mode should support Postgres-backed control state and pluggable durable event infrastructure.

6.10 Protocol Adapter Mesh

Adapter Role Why it belongs Priority
MCP client Tool discovery, tool invocation, session management MCP is the fastest path to a broad tool ecosystem P0
A2A connector Remote agent discovery and delegation Extends Crawfish beyond one framework and one trust domain P1
Local subprocess adapter Isolated coding agents and automation workers over stdin/stdout or RPC Important for coding use cases and tool sandboxing P1
Custom transport plugins Domain-specific tools and interop backends Preserves extensibility without bloating core P2

The key design choice is normalization. Each adapter should expose a capability descriptor with verbs, risk class, cost class, latency class, mutability, and required approvals. This is how Crawfish keeps policy consistent even when execution crosses MCP, A2A, and local adapters.

6.11 Observability And Governance

Tracing alone is not enough. The runtime must emit operational meaning, not just call trees.

  • Trace model: spans for task, agent turn, tool call, approval gate, scheduler decision, and lifecycle transition.
  • Metric model: deadline hit rate, budget overrun rate, recovery success rate, active degraded agents, approval wait time, checkpoint replay count, token and dollar burn per successful task.
  • Audit model: who requested what, which policy applied, why fallback happened, who approved writes, what artifact was produced.
  • Governance model: tenant boundaries, secret references, tool allowlists, mutable action logs, and retention controls.

6.12 Security Model

Area Requirement
Secrets Secrets must be referenced, not embedded. They should be mounted into tool sessions only when contract and policy allow it.
Tool permissions Tools are declared with risk class and mutation scope. Runtime policy decides whether proposal-only, approval-gated, or autonomous execution is permitted.
Workspace isolation Coding workloads require per-task or per-agent workspace isolation plus file locking or branch isolation for mutable operations.
Approval gates Human approval must be a first-class runtime state, not an application afterthought.
Tenancy Tenant IDs and data-zone constraints must propagate through tasks, checkpoints, and remote delegation.

11. Go-To-Market Narrative

11.1 Launch Story

The launch narrative should not claim "we orchestrate agents better" in the abstract. It should claim something more concrete and more defensible: most frameworks help you build agent logic; Crawfish helps you operate agent systems.

11.2 Hero Demo

The hero demo should be a small engineering or operations fleet, not a generic chatbot. Example: repo indexer, repo reviewer, CI triage agent, and release prep agent running under one supervisor, with visible degradation, approvals, and recovery.

11.3 Open Source Strategy

  • Open-source the core runtime, manifest model, MCP integration, CLI, and telemetry schema.
  • Make the operational story obvious from day one: inspectable state, visible contracts, clear logs, and restart recovery.
  • Document one local mode and one production reference deployment so users see a path from laptop to team use.

12. Risks And Mitigations

Risk Impact Mitigation
Over-engineering High Protect the agent authoring surface. Keep manifests small. Move complexity behind the runtime and policy compiler.
Protocol churn Medium Keep adapters thin and normalize capabilities and contracts above them.
Weak differentiation against workflow engines Medium Focus the narrative on lifecycle, degraded operation, tool safety, and policy-preserving interop.
Security incidents in always-on mode High Ship approval gates, scoped tools, secret refs, audit logs, and read-only degraded mode early.
Complex mental model for users High Provide strong templates, clear defaults, and a progressive path from single agent to fleet.

14. Final Recommendation

Crawfish should position itself as the runtime layer the current agent ecosystem is missing. The key is not to promise "smarter agents." The key is to promise operable agents.

The strongest version of this product is one that keeps the authoring model simple while making the runtime operationally rich: managed lifecycle, action-based durability, execution contracts, protocol-native adapters, and built-in governance. If those pieces come together cleanly, Crawfish will not look like another orchestration framework. It will look like the control plane agent systems have been missing.
